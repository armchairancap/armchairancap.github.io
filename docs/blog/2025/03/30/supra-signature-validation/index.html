Supra signature validation without the SDK

March 30, 2025 · 6 min read
ArmchairAncap
  • Intro

  • Cryptography
  • Sign and validate
  • Take-aways and next steps

Intro​

How to sign and validate a message using your wallet private key from the CLI?

I blogged about this topic in the context of xx Network and I need this for Supra Network as well.

Supra CLI​

I haven't spent much time playing with Supra CLI, but recently I got my hands on the Docker image v8.0.2 (which is currently the main way Supra distributes the CLI package).

You can read about getting started in the CLI section of the Supra Documentation. You download the Docker container and create a profile, which stores your settings similar to below (line breaks added for easier viewing).

$ supra profile -l 
Enter your password: 
-------------------------- Profile List ---------------------------------
	(*) deftest
CliProfile { account_address: "0xf63b51af98a1ff6b469bb5b8e9fa4b81733b2ef79234c4bffd949bf53f6579dc", 
            ed25519_public_key: 02d0b5b6226d8b9606aedac673de9591f34745f1e38d2f3805067dba2ed5b1a5, 
            rpc_url: "https://rpc-testnet.supra.com/", 
            faucet_url: "https://rpc-testnet.supra.com/", 
            chain_id: 6 }
-------------------------------------------------------------------------

The above is my default profile for testnet. Where's the private key? You need -l -r to include the private key as well.

Sign and validate​

So, how to sign stuff? ("Stuff" means stuff - anything that can be fed via STDIN or referenced as a file).

$ supra profile sign --profile deftest --stdin reputable.supra
Enter your password: 
ContentSignature { signature: "c36472c483adbc6d311234984debd9947fbf25d36539183fe4a100da335f7839fa3cb3c276a82a025b5a1464a459aeacec5383a7c6dffa387fce08a7b0e1c203" }

If I had an image such as img.png, I'd do --file ./img.png.

Note that if a profile has multiple addresses, I'd also provide the account address.

$ supra profile sign -h
Sign the payload

Usage: supra profile sign [OPTIONS]

Options:
      --profile <PROFILE>                        Profile to use from the CLI config
      --private-key-file <PRIVATE_KEY_FILE>      Signing Ed25519 private key file path
      --private-key <PRIVATE_KEY>                Signing Ed25519 private key
      --account-address <ACCOUNT_ADDRESS>        
      --rpc-url <RPC_URL>                        
      --faucet-url <FAUCET_URL>                  
      --sender-account <SENDER_ACCOUNT>          Sender account address
      --file <FILE>                              
      --stdin <STDIN>                            
      --expected-signature <EXPECTED_SIGNATURE>  
  -h, --help                                     Print help (see more with '--help')

In any case, a signature is generated by signing stuff with the Ed25519 private key (in Supra CLI profile, it's named ed25519_secret_key) that belongs to the profile and address you provided.

Now the fun part: there's just supra profile sign, but no supra profile validate. Oops!

$ supra profile -h
Manage Supra CLI profile

Usage: supra profile <COMMAND>

Commands:
  list, -l, --list             List public keys of all profiles.
  activate                     Activate the profile.
  new                          Creates a new profile.
  generate-validator-identity  Generate validator identity file.
  remove                       Remove existing cli profile
  modify                       Modify the profile attributes
  sign                         Sign the payload
  help                         Print this message or the help of the given subcommand(s)

I suppose this is when they tell you "just use the SDK". Yeah, no, thank you.

I don't mean that it wouldn't work, but rather that then I'd have to use the SDK which is precisely what I don't want.

I want something lightweight that works in Python. And there's no officially supported Supra SDK for Python (there's this fork, but it's not supported and who knows if it even works. I haven't checked whether it contains signature validation functions).

So, it's back to that RTFM link above ("generic" Python Cryptography module). Using that approach, you can create a script that uses your private keyh to sign stuff in Python.

You can make it so that you provide a private Ed25519 key and string to sign, sign and then validate.

PRIVKEY="xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
$ ./sign_message.py ${PRIVKEY} reputable.supra

Signature:
f506d305fb4026d9822b09a547c30443f299a14a583e4b95b05466fe60d7d59b6dc7f2b7612fcb0afd19ec285ec246e59556c5590655938ad105b402fd3ab400

Signature is valid.

That's great, right? Not really.

I got a different signature using the same inputs that I used with Supra CLI!

# f506d305fb4026d9822b09a547c30443f299a14a583e4b95b05466fe60d7d59b6dc7f2b7612fcb0afd19ec285ec246e59556c5590655938ad105b402fd3ab400 # Python 
# c36472c483adbc6d311234984debd9947fbf25d36539183fe4a100da335f7839fa3cb3c276a82a025b5a1464a459aeacec5383a7c6dffa387fce08a7b0e1c203 # Supra CLI

And, even worse, if I validate the Supra CLI signature using my Python script, it fails.

That means we didn't sign it exactly the same way.

If I sign and validate signature ("end to end") in Python, it works. But I can't validate Supra CLI signatures using the right public key.

note

I spotted this part which hints at deterministic signatures, but I didn't get anywhere with that lead.

How it possible to get different result? Very easily. The Supra documentation doesn't have those details yet, so to start, one would have to read this and check if that's how it works on Supra...

Take-aways and next steps​

Supra CLI lets me sign, but not validate.

Python Cryptography module lets me do both, but I can only validate what I signed with Python, and not what Supra CLI signed.

info

The Python Cryptography approach isn't completely useless: if I sign and validate using it, it does work.

The reason to not like it is getting a private key out of your wallet (or CLI profile) is risky (and annoying). It can be fine in some isolated batch-processing environment where all keys are stored in a database, but it's not a good idea for regular use and it's completely impractical for StarKey use.

My use case happens to be batch processing, but not of the kind that can work: my input is a Supra CLI-signed signature which means I can't validate it with my Python approach (unless I find a way to fix it).

Another shortcoming - which may or may not bother you - is that the Python approach doesn't give you the wallet address. If you scroll up to see Supra CLI profile list output, you'll see that account_address is different from ed25519_public_key.

Even if you perform end-to-end signing and validation in Python, you won't know the wallet address without additional steps or more information provided by the user. If you need to transfer coins or do something else, you'll need additional steps to compute account address from ed25519_public_key (doesn't look hard, e.g. see this, but has to be done if you need it and it'd better generate the address that's in wallet, in terms of derivation path). I tried SHA256-hashing ed25519 public key and did NOT get the correct address...

I'm happy to see that people struggle with this even if they use Aptos (or Supra) SDK, so I'm probably not the dumbest enthusiast out there.

If I figure out how to do this and remember to update this post later, I'll do it.

My next attempt may be with Aptos Python SDK on Aptos Network and, if that doesn't work, maybe I'll fall back to NodeJS CLI (and SDK).
