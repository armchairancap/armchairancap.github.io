Introduction​

In my Hacktoberfest 2024 PoC for token-gated access on xx Network I wrote about restricting access to Web apps based on criteria such as the amount of XX token or kind and quantity of asset a wallet holds.

At the time I didn't look at NFTs, because they're not even exposed in xx Network wallet and I wasn't sure if those even worked.

Well, today I took a look and:

  • The API is there and works
  • The official xx Network wallet indeed doesn't expose them in any way
  • NFT pallet in xx Chain code 0.2.6 is based on pallet-nfts v4.0.0

So, in theory at least, it should be possible to use them as long as you have your own application or don't need a Web UI.

I decided to spend some time on that this afternoon.

NFT concepts​

It's a Substrate thing, similar across different chains. Note that some are more advanced and NFT-focused, and there you can do a lot more.

  • NFTs come in "collections". A collection is identified by an ID, and has a bunch of properties
  • Each NFT belongs to a collection. Example: collection ID 10 may have NFTs ID 1, 2, and 3
  • NFTs can be minted, priced, transferred, bought, burned. Soul Bound NFTs are those that cannot be transferred once they're minted to a wallet

On xx Network, since there's no user-facing Web UI where you can see anything. There's a simple "NFT" page that's always empty and doesn't show any collections or holdings, which is confusing. You can own, but can't see if you own any collections or NFTs.

For more on Polkadot SDK-based NFTs, see this page on the Polkadot Wiki: https://wiki.polkadot.network/docs/learn-nft-pallets.

NFT operations​

This is a partial list of NFT-related operations available on Polkadot (source: Polkadot NFT documentation).

Permissionless​

These are not limited to the owner.

  • create: Create a new collection by placing a deposit
  • mint: Mint a new item within a collection (when the minting is public)
  • transfer: Send an item to a new owner
  • set_price: Set the price for an item
  • buy_item: Buy an item
  • pay_tips: Pay tips, could be used for paying the creator royalties
  • create_swap: Create an offer to swap an NFT for another NFT and optionally some fungibles
  • cancel_swap: Cancel previously created swap offer
  • claim_swap: Swap items in an atomic way

Permissioned​

These are "admin" level actions for the issuer.

  • destroy: Destroy a collection. This destroys all the items inside the collection and refunds the deposit
  • force_mint: Mint a new item within a collection
  • burn: Destroy an item within a collection
  • lock_item_transfer: Prevent an individual item from being transferred
  • unlock_item_transfer: Revert the effects of a previous lock_item_transfer
  • clear_all_transfer_approvals: Clears all transfer approvals set by calling the approve_transfer
  • lock_collection: Prevent all items within a collection from being transferred (making them all soul bound)
  • lock_item_properties: Lock item's metadata or attributes
  • transfer_ownership: Alter the owner of a collection, moving all associated deposits. Ownership of individual items will not be affected
  • set_team: Alter the permissioned accounts of a collection
  • set_collection_max_supply: Change the max supply of a collection
  • update_mint_settings: Update the minting settings for collection

Permissioned for metadata​

Same as above, but for metadata.

  • set_attribute: Set a metadata attribute of an item or collection
  • clear_attribute: Remove a metadata attribute of an item or collection
  • set_metadata: Set general metadata of an item (e.g. an IPFS address of an image url)
  • clear_metadata: Remove general metadata of an item
  • set_collection_metadata: Set general metadata of a collection
  • clear_collection_metadata: Remove general metadata of a collection

Force (governance) operations​

Also "admin" level operations for the issuer, mostly related to "override" operations.

  • Create a new collection (the collection ID can not be chosen)
  • Change collection's owner or configuration
  • Set an attribute

Getting started with NFTs​

To make my life less miserable:

  • I got two wallets (addresses), and created on-chain ID for the "NFT issuer" wallet
  • I got around 150 XX handy (20 XX for on-chain ID for the "NFT issuer", 100 as collection (issuance) deposit for one NFT collection, 10 for each NFT (I issued two))

As wallet addresses are hard to remember, let's say:

  • Issuer: ArmchairAncap-Test-Wallet (6Z...)
  • Buyer: The-Dude aka "six-ace" (6aCE...)

Recap​

So far, this is what I got:

  • ArmchairAncap-Test-Wallet creates a collection with the ID 0, up to 500 of them, priced at 1 XX
  • ArmchairAncap-Test-Wallet mints two NFTs (IDs 1 and 2):
    • ID 1 minted to The-Dude
    • ID 2 issued to himself
  • ArmchairAncap-Test-Wallet plays with settings, and makes just one big change - creates a year-end special for The-Dude: sets the price of NFT ID 2 to 0.5 XX

What's minted to others gets "sent" to those wallets right away.

When issuer only can mint, what's issued to self seemingly cannot be bought without the issuer's specific "offer" to a 3rd party wallet such as The-Dude. I haven't verified this, but it seems that's how it works.

Buy an NFT​

And the last screenshot shows how to buy that NFT ID 2 from the issuer who made it 50% cheaper than other items in the collection, so The-Dude buys it. (Reference: purchase extrinsic)

I haven't tried using the API, but buying that NFT item for 0.5 XX would go similar to this:

substrate.compose_call( 'Nfts', 'buy_item', {'bid_price': 500000000, 'collection': 0, 'item': 2 })

After purchase, this item ID 2 is in the possession of The-Dude aka six-ace:

>>> substrate.query( 'Nfts', 'Item', [0, 2])

There are similar little details that should be checked further, such as the fact that the issuer may not be the only person who can mint NFTs in a collection (!), so make sure to read the docs before using these in production.

What can we do with NFTs?​

Many of the things everyone else does.

See that post on token-gated access for some ides as well as the PoC app and its source code.

Personally, now that I know this API works and how it compares to xx Chain assets, I think NFTs are even better for mebership- and subscription-like use cases than assets!

There's no need to buy NFTs "off the market" like assets, which is a major advantage. You can also set discounts, named buyers, and more.

I'm not interested in popular NFT use cases (memecoins, games and such), so the fact that the xx Network Wallet has nothing for that doesn't bother me at all. xx Network assets are much better in this regard - you can see which assets exists, and how holds how many - but doing anything with them is difficult and especially buying and selling.

Other chains may have a more advanced NFT pallet, but I'd say xx Chain doesn't even need those newer pallet. Other chains need apps that use NFTs, but xx Chain already has a good one - it's called Haven.

While one may NFT-gate a Web site, you can't NFT-gate a Haven channel because the same channel (Space) can be accessed from any instance. But: we can ask Haven users to sign their wallet address using their wallet, and then easily check if the user owns an NFT (or some quantity of an xx Network asset).

Next steps​

xx Chain can do a lot more with NFTs than the spartan UI suggests - it's more than enough for some very interesting use cases.

It's shocking that no one has done anything with these so far, in over 1100 days since the chain launched.
