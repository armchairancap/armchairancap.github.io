Supra signature validation without the SDK

March 30, 2025 · 6 min read
ArmchairAncap
  • Intro

  • Cryptography

Intro​

How to sign and validate a message using your wallet private key from the CLI?

I blogged about this topic in the context of xx Network and I need this for Supra Network as well.

Supra CLI​

I haven't spent much time playing with Supra CLI, but recently I got my hands on the Docker image v8.0.2 (which is currently the main way Supra distributes the CLI package).

You can read about getting started in the CLI section of the Supra Documentation. You download the Docker container and create a profile, which stores your settings similar to below (line breaks added for easier viewing).

$ supra profile -l 
Enter your password: 
-------------------------- Profile List ---------------------------------
	(*) deftest
CliProfile { account_address: "0xf63b51af98a1ff6b469bb5b8e9fa4b81733b2ef79234c4bffd949bf53f6579dc", 
            ed25519_public_key: 02d0b5b6226d8b9606aedac673de9591f34745f1e38d2f3805067dba2ed5b1a5, 
            rpc_url: "https://rpc-testnet.supra.com/", 
            faucet_url: "https://rpc-testnet.supra.com/", 
            chain_id: 6 }
-------------------------------------------------------------------------

The above is my default profile for testnet. Where's the private key? You need -l -r to include the private key as well.

Cryptography​

This is not my area, but we have search engines (and more recently, AIs).

To get started, I referenced [1]:

What we can see here is raw keys, which - as we shall see later - is the same thing that Supra CLI shows us. This hopefully means we can use the same tools to get the same result.

I then proceeded with RTFM to get it done.

Reference: [1] https://asecuritysite.com/openssh/openssh2

Sign and validate​

So, how to sign stuff? ("Stuff" means stuff - anything that can be fed via STDIN or referenced as a file).

$ supra profile sign --profile deftest --stdin reputable.supra
Enter your password: 
ContentSignature { signature: "c36472c483adbc6d311234984debd9947fbf25d36539183fe4a100da335f7839fa3cb3c276a82a025b5a1464a459aeacec5383a7c6dffa387fce08a7b0e1c203" }

If I had an image such as img.png, I'd do --file ./img.png.

Note that if a profile has multiple addresses, I'd also provide the account address.

$ supra profile sign -h
Sign the payload

Usage: supra profile sign [OPTIONS]

Options:
      --profile <PROFILE>                        Profile to use from the CLI config
      --private-key-file <PRIVATE_KEY_FILE>      Signing Ed25519 private key file path
      --private-key <PRIVATE_KEY>                Signing Ed25519 private key
      --account-address <ACCOUNT_ADDRESS>        
      --rpc-url <RPC_URL>                        
      --faucet-url <FAUCET_URL>                  
      --sender-account <SENDER_ACCOUNT>          Sender account address
      --file <FILE>                              
      --stdin <STDIN>                            
      --expected-signature <EXPECTED_SIGNATURE>  
  -h, --help                                     Print help (see more with '--help')

In any case, a signature is generated by signing stuff with the Ed25519 private key (in Supra CLI profile, it's named ed25519_secret_key) that belongs to the profile and address you provided.

Now the fun part: there's just supra profile sign, but no supra profile validate. Oops!

$ supra profile -h
Manage Supra CLI profile

Usage: supra profile <COMMAND>

Commands:
  list, -l, --list             List public keys of all profiles.
  activate                     Activate the profile.
  new                          Creates a new profile.
  generate-validator-identity  Generate validator identity file.
  remove                       Remove existing cli profile
  modify                       Modify the profile attributes
  sign                         Sign the payload
  help                         Print this message or the help of the given subcommand(s)

I suppose this is when they tell you "just use the SDK". Yeah, no, thank you.

I don't mean that it wouldn't work, but rather that then I'd have to use the SDK which is precisely what I don't want.

I want something lightweight that works in Python. And there's no officially supported Supra SDK for Python (there's this fork, but it's not supported and who knows if it even works. I haven't checked whether it contains signature validation functions).

So, it's back to that RTFM link above ("generic" Python Cryptography module). Using that approach, you can create a script that uses your private keyh to sign stuff in Python.

You can make it so that you provide a private Ed25519 key and string to sign, sign and then validate.

PRIVKEY="xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
$ ./sign_message.py ${PRIVKEY} reputable.supra

Signature:
f506d305fb4026d9822b09a547c30443f299a14a583e4b95b05466fe60d7d59b6dc7f2b7612fcb0afd19ec285ec246e59556c5590655938ad105b402fd3ab400

Signature is valid.

That's great, right? Not really.

I got a different signature using the same inputs that I used with Supra CLI!

# f506d305fb4026d9822b09a547c30443f299a14a583e4b95b05466fe60d7d59b6dc7f2b7612fcb0afd19ec285ec246e59556c5590655938ad105b402fd3ab400 # Python 
# c36472c483adbc6d311234984debd9947fbf25d36539183fe4a100da335f7839fa3cb3c276a82a025b5a1464a459aeacec5383a7c6dffa387fce08a7b0e1c203 # Supra CLI

And, even worse, if I validate the Supra CLI signature using my Python script, it fails.

That means we didn't sign it exactly the same way.

If I sign and validate signature ("end to end") in Python, it works. But I can't validate Supra CLI signatures using the right public key.

note

I spotted this part which hints at deterministic signatures, but I didn't get anywhere with that lead.

How it possible to get different result? Very easily. The Supra documentation doesn't have those details yet, so to start, one would have to read this and check if that's how it works on Supra...

Take-aways and next steps​

Supra CLI lets me sign, but not validate.

Python Cryptography module lets me do both, but I can only validate what I signed with Python, and not what Supra CLI signed.

info

The Python Cryptography approach isn't completely useless: if I sign and validate using it, it does work.

The reason to not like it is getting a private key out of your wallet (or CLI profile) is risky (and annoying). It can be fine in some isolated batch-processing environment where all keys are stored in a database, but it's not a good idea for regular use and it's completely impractical for StarKey use.

My use case happens to be batch processing, but not of the kind that can work: my input is a Supra CLI-signed signature which means I can't validate it with my Python approach (unless I find a way to fix it).

Another shortcoming - which may or may not bother you - is that the Python approach doesn't give you the wallet address. If you scroll up to see Supra CLI profile list output, you'll see that account_address is different from ed25519_public_key.

Even if you perform end-to-end signing and validation in Python, you won't know the wallet address without additional steps or more information provided by the user. If you need to transfer coins or do something else, you'll need additional steps to compute account address from ed25519_public_key (doesn't look hard, e.g. see this, but has to be done if you need it and it'd better generate the address that's in wallet, in terms of derivation path). I tried SHA256-hashing ed25519 public key and did NOT get the correct address...

I'm happy to see that people struggle with this even if they use Aptos (or Supra) SDK, so I'm probably not the dumbest enthusiast out there.

If I figure out how to do this and remember to update this post later, I'll do it.

My next attempt may be with Aptos Python SDK on Aptos Network and, if that doesn't work, maybe I'll fall back to NodeJS CLI (and SDK).

About some special SupraNS (SNS) names

March 29, 2025 · 5 min read
ArmchairAncap

Intro​

Who should own what digital asset isn't a new topic - it's just a version of the same situation with DNS domains - and there are many articles and blog posts on best ways to deal with it.

In this post I'll explain my approach to this problem on Supra Name Service (SNS) and explain why I think it's good.

Among the SupraNS names I own, some are "special" in the sense that they may be non-generic names used by some entity in the real world.

I figured those were going to be registered by others anyway, so I thought it prudent to register them myself.

note

I also own a few names that I registered as "community service" - I wanted to take them off the market and make sure they can't be used.

Those aren't discussed in this article and I won't sell them. Maybe one day they'll expire on their own.

What follows is my Q&A on how I plan to use and wortk with "special" SNS names.

Is there a list of "special" SupraNS names I own?​

Not yet.

The "special" SNS names I own currently aren't listed on the SupraNS site where my SupraNS names are available for sale.

The reason is I don't want to make them available to random buyers. I would like to make them available to users who should own them.

How can one find those "special" SNS names?​

If you've asked yourself that question or if the question above makes sense to you, then you realize there's a discoverability problem. Congratulation!

You can look up a name at https://suprans.id/names and if you notice the wallet address that owns the name is the same one that SupraNS mentions, then I have it. Next, go to SupranS.name and search for it. If it's listed as "special" or not listed at all it's "special".

If you run a project or business that uses one of the "special" SNS names I own, reach out to me on X.

I can transfer them for a small fee that covers my cost of labor.

What happens if no one steps forward by the time such-and-such name expires?​

I can let that SNS expire in which case the project or business that wants it has to race against others who may want to acquire that same name for resale or own use.

That approach seems bad, so I plan to auction such names starting in late 2025 or early 2026 (i.e. 2-3 months before they expire).

How is auctioning them better than letting them expire?​

As explained above, letting them expire doesn't mean the "right" user (if that can be determined at all!) would succeed to get that name.

It's equally likely that a competitor or squatter gets the name. If three people compete for registering the domain, each person's odds are 1/3.

The second problem is discoverability/visibility mentioned earlier. How does a person even learn of Supra to begin with?

Someone who realizes a "special" SNS name they want is registered today should be be relieved to find that it is not available for sale in open market and - assuming they represent a business that makes use of such name - they could easily get it right away.

But, it is almost certain that many users who'll one day want that "special" SNS name won't even learn of SupraNS before my special SNS names expire in March 2026.

If these special names aren't on the market (as they aren't on SupraNS.name now), they have no visibiliy. It's hard to discover them and that makes it less likely that those who may want them will become aware of them in 2025. Because of that:

  • If they reach out in 2025, they'll get them affordably and easily (as long as they represent a business or project that ought to own the SNS)
  • If they by some chance don't pay attention in 2025, they would probably appreciate to learn in early 2026 that the name they may want appeared at some auction marketplace and get a chance to buy them at the price they can afford

In order to get a second chance and gain visibility before they expire, these special names should hit the market.

The worst case: a name expires and gets squatted by a "hard-ass squatter" who wants US$20,000 for it or - even worse - an impostor, scammer or competitor.

The "best" case: a name expires and no one wants to use it, so you get a chance to register it in days or weeks after that. If your "special" SNS name is not worth the Dexlyn SNS registration fee, that might work.

Take-aways​

Many SupraNS brands have been squatted and some may be squatted by scammers or competing businesses.

I don't know if Dexlyn plan to implement "conflict resolution" procedures available for Internet DNS TLDs.

I think my two-phase approach is ethical, fair, effective, and works well.

Intro​

Some people who advocate DEXes are decentralization fanatics, others are conspiracy theorists, and some "all of the above".

There are also pragmatists in this group. In this post I share my pragmatic arguments in favor of moving your active stash to a DEX.

Note that this is specific to Supra and my own purpose, which is to see the project grow and, with it, the value it provides to its ecosystem. More value means more growth and a higher coin price. I view Supra as my "enterprise" blockchain holding: I'd like it to see grow as a normal for-profit business would.

note

I elaborate because I also hold xx coin which is more of a "cause coin" for me and where I'm not pragmatic in terms of the token price, earnings and such.

In the case of xx coin, I'm interested in "economically-sustainable private messaging" and have very different concerns.

In the case of Supra, I don't think centralization is bad. I simply don't like how these particular unregulated guys run their business.

Another special circumstance is Supra is a good L1 platform for DEX markets, so I also have - all Supra holders and stakers should have - a conflict of interest in their mind.

Current Supra-listing CEXes​

I don't expect to add anything new here, as cryptocurrency DEXes have been around for a decade or more, and pragmatic arguments against CEXes have been around longer.

Briefly, from the viewpoint explained above (meaning: pragmatic, Supra-specific arguments from an enterprise POV), this is what's wrong with CEXes (especially the unregulated ones):

  • Lack of transparency: this is really the worst part
  • Poor business practices: we don't know who and how (why, because no transparency) not only in terms of transparency, but also in terms of governance
  • Leverage (Futures): this isn't bad in itself (it also exists on regulated CEXes, and on DEXes), but I think it's a major tool used in market manipulation because it's likely used against retail users on unregulated CEXes

Regarding the lack of transparency, folks on X like to make fun of crypto/DeFi fans saying cypherpunks, who claimed zero regulation was good, start whining when their altcoin gets smashed on the same markets. I agree with them:

  • I may decide to participate in a zero-regulation market (a DEX, for example) and I won't whine if lose
  • Zero-regulation (or "extremely lax" regulation) CEX could work if these exchanges voluntarily shared data that should be shared. But they don't share anything of substance, while regulated CEXes probably collect and share too much, which increases costs for no benefit to users. There's nothing "in-between" which is a shame
info

CEXes also have some good sides and advantages. For example, most of these are still true:

  • Low latency, very suitable for frequent trading
  • Better features for advanced traders (such as leverage, automation, APIs, clients)
  • Convenience (in some cases)

CEXes still command close to 80% of crypto spot trading and there's nothing wrong with using them. People say "not your private keys not your coins", but I would guess more people lost their coins in self-custody than at CEXes.

Now I'll give some anecdotal examples of how unregulated CEXes suck.

Governance​

Transparency is sometimes legally mandated, but for unregulated exchanges it's merely a choice.

There's nothing that prevents them from busting manipulators, reporting money launderes, and such. From what I can tell the unregulated CEXes where Supra is listed are doing the bare minimum.

Another area of governance is operations. If you work for a regulated FSI player you're simply not allowed to do certain things the weird or wrong way. There are checks and checks of checks.

For example, using plain text passwords could be grounds for dismissal, and you'll get discovered because in the regulated world Operations are not in charge of Security - there's a Security team who audits and checks Operations. But the ByBit guy transferred $1.5B to the North Korean hackers because ByBit used a DIY workflow "invented" by ByBit where security practices failed. Had they followed established practices in place in the UK or US, they'd work slower, safer and more expensively. They probably didn't like the speed and cost of the more "regulated" approach and thought they were smarter... Fortunately ByBit - and with it its Supra users - survived (for now).

This argument applies to DEXes as well (were bugs may go undetected due to corner cutting in development, operations or auditing), but it's not an argument against regulated CEXes.

Unfortunately, no Western (regulated) CEX list Supra at this time, so there's no "better" CEX we can go to.

One example of poor transparency is market making (or manipulating... - with unregulated CEXes you can't even know which it is).

Consider this screenshot of trades in SUPRA/USDT (Spot) taken yesterday.

Again, the problem here isn't that there's a market "maker". The problem is that they're a "taker". They actively trade.

If you make a market, you create buy/sell orders that others "take"" (by issuing matching sell/buy orders). If there's no genuine activity, market maker's orders just sit there for the time when a user comes, to have liquidity to sell or buy into. Market makers aren't supposed to impact the price.

Seeing a lot more volume when they're not day-tripping means when they're gone there's no trading. Obviously, this means bots are rampant. These MM guys engage in wash trading (to create appearance of activity) and various forms of market manipulation all deemed illegal in regulated markets.

And, related to transparency, you don't even know who's MM-ing, who's paying them, and if they're actively trading against exchange's retail users. But, you'd probably be right if you assumed the worst (such as, that these deregulated CEXes don't care what kind of MM-ing - making or manipulation - goes on).

I've never heard of anything like tha thappening at any of the exchanges that list Supra and I assume "reasonably crooked" MMers operate even on well-run unregulated exchanges.

Some "crypto people" told me I'm nit-picking, and "it's all MM". That is nonsense. If you take orders, meaning create matching bids for existing buy or sell orders, you directly impact the price and secondly, you earn or lose from such trades. And because of that, you'll obviously try to profit, and the result is you're now trading against CEX users. You can't profit unless they lose. So what, you may ask?

  • The CEX can obviously see (see the screenshot at the top) that you're taking orders i.e. trading against users, which is the first problem
  • The second problem is they can also easily see you're entering short positions in one place (such as Spot) and long (such as Futures) in another, for the same pair. That is, you're clearly there to manipulate the price by creating artifical price swings. You're not either selling or buying

What is the difference between a Market Maker and a regular user with an AI bot? If you're a Market Maker, not a Manipulator, no regulated exchange will let you do that. Even Binance (not the gold standard by any means) won't allow it. From the Binance link above:

General reminder that all projects’ authorized market makers on Binance must comply with the principals and rules as follows:

  • Place orders for both bid and ask
  • Ensure orders remain for a certain amount of time, and prevent market disruption from high-frequency order placement and cancellations

When a Market Maker violates the rules (in the Binance case), what happens is although they're authorized (and usually paid by someone) to make markets, provide liquidity and build trust, they start acting on behalf of other "customers" (they trade on own behalf or maybe even on behalf of other manipulators, to assist others to manipulate the market they make). Extremely bad!

Although it may seem such dishonest Market Makers could simply become general exchange users and do whatever they want, that isn't the case. Regulated CEXes do not allow market manipulation explained above, even if it's done by regular users.

Perpetual Futures aka Perps​

Perpetual futures are basically entries in a database which is a closed zero-sum system where there is no Supra involved at all.

In fact, there's no USD involved either! Yes, it says SUPRA/USDT, but understand that this Perps market doesn't need USDT just like it doesn't need SUPRA.

  • A DB is created, and named after a pair such as SUPRA/USDT
  • To play, gamers deposit USDT "at the entrance" in order to be able to enter and play, but just like Supra (the other side of this pair), the Perps DB does not "use" USDT. It's just a database
  • Next, players play against each other, and the survivors "unlock" thei USDT deposit and collect deposit of other users who've lost, while the CEX retains trading revenues
  • Players could deposit collateral in gemstones, Litecoin or something else. USDT is simply what's most convenient in this case because it maps 1:1 to a side in this pair
  • Players cannot take Supra out of this market, because there's no Supra, as we've said earlier

Next, realize that Perps provide leverage - up to 50x.

There are two interesting things about this.

The first is, players can deposit 10 USDT "at the entrance" and play with 500 USDT. This speeds up the CEX'es "time to fleecing". Notice how it also makes $500 available to CEX interest-free. It's a closed system so as long as they liquidate overleveraged positions on time, there's no chance for CEX to lose here. This is obviously much better (for CEXes) and worse for their customers. Instead of hesitating to send users to Futures section (where more are likely to lose money), they employ "dark UI patterns" to attract users to Perps. MEXC won't even show Spot in its top menu once you enter Futures!

Point 2: with a 50x leverage you can buy, but also sell, 50x more Supra than you could normally sell. That means Perps are the place to go to smash a project or simply cause price-impacting swings in Spot SUPRA/USDT as "the tail is wagging the dog".

But it's even worse than that! We said "50x more Supra than you could normally sell", but in order to sell you'd have to buy. With Perps not only do you can sell 50x more, but you don't have to buy any at all. You deposit USDT at the entrance, lever up 40x, smack Supra, and exit with more USDT in your pockets.

You may wonder why this doesn't equally work for buyers.

In truth it does, but that doesn't make this better. If the marketplace owner determines you're trading in a way meant to manipulate Spot prices - for example you're shorting in Perps with 40x and and have buy orders in Spot - they should step in and do something about that. These cowboy CEXes do not.

I think unregulated cowboy exchanges don't do anything about it. If anything, they encourage it.

DEXes​

I've used the early ones on bitcoin network and it was bad! I remember how we'd see a sell order, submit our matching buy order and then wait, and wait, awnd wait - usually for more than 1 hour - before we could see our order on chain (6 confirmations, remember?). Anyone who saw your order in mempool could front-run you if they wanted to pay a slightly higher transaction fee.

Things are better now. This aren't "latest and greatest", but they're recent enough (2025). The first is DEX share in Spot volumes.

Perps are harder, as mentioned earlier, but both blockchains and DEX tech are getting better.

CEXes have noticed. See this, for example:

We need to rethink our listing process at @coinbase

There are several reasons, but in January commissions from memecoin trading were huge and CEXes just couldn't keep up - all the action was happening on Solana, basically.

But it's not just memecoins, it's also "traditional" coins which probably don't last long enough and CEXes don't want to miss out.

DEXes don't solve all the problems with unregulated CEXes mentioned above, and some maybe aren't even solvable.

For example, leveraged perpetual futures are also available on on-chain DEXes. But at least you don't lend your own funds to a CEX interest-free to use them against you.

DEX benefits to you and Supra ecosystem​

I'll keep this simple because it ought to be obvious and it was explained at the top:

  • Better transparency, fairness and more disclosure than unregulated CEXes
  • Convenience, performance and features are very close - it's good enough for non-professionals
  • Supra-based DEXes trade on-chain, chain earns revenue, more people buy Supra to pay for those fees
  • Supra-based DEXes need liquidity. You can stake DEX pools and make money doing that

It's also a way to distribute risk for your "active" Supra stash - part of your tokens that moves more often, as opposed to staked Supra that is likely quite static. You may do something like this:

  • 1/3 of active stash on a regulated CEX if you feel the need to day trade from time to time (some people prefer unregulatd; I do too, but not any of the current ones)
  • 1/3 in Staking or LP Staking on Supra-based DEX number 1 (e.g. Dexlyn)
  • 1/3 on Staking or LP Staking on Supra-based DEX number 2 (e.g. Atmos)

Slippage and fees​

Everyone knows Supra is fast and transaction costs are very low, Supra's DORA (Distributed Oracles) provides up to date price feeds quickly, but one "sensitive" bit about DEXes is slippage and fees.

There isn't enough independent detailed information available to compare, but anecodtally:

  • A low cost, unregulated CEX may have a 0.05% "taker" fee (example here)
  • Dexlyn may have a 0.3% commission and "slippage" that's to be kept at below 0.5% (default permitted slippage setting)

The problem is we don't really know how it behaves (slippage). From my experience with smaller (below $1,000) amounts, Dexlyn worked fine when I compared the price from that UI with CEX prices in another browser.

info

Maybe this doesn't work that great with larger orders, but I haven't seen any evidence, so I can't conclude much from unsubstantiated "concern trolling".

Yes, perhaps, if you Swap $10,000 today, you'd be hit with 2% slippage. It's theoretically possible. But don't do $10K at a time if you notice at $2K it's already higher than 0.5%!

For normal retail use swapping on Dexlyn right now will cost you up to 0.8% (up to 0.5% slippage plus fixed 0.3% commission). It can be below 0.5% - less than you pay at a low-cost deregulated CEX!

This may change for better or worse, but it's quite simple and low-risk: don't use a crap DEX you've never heard of and run small tests before ramping up like a degen! Transactions take seconds, so I'm sure you can spend 30 seconds trying with $10, $50, and $100 before you try with $500. They also have a Testnet to try with fake coins first.

Supra Network makes it possible to address most of those problems, so while slippage has historically been a week spot for DEXes, it's not supposed to happen on Supra if DEXes make good use of those L1 features.

The current lot of unregulated exchanges that run Supra Spot market are in my opinion not good hombres, folks.

SupraNS (SNS) names with hexadecimal symbols

March 19, 2025 · 3 min read
ArmchairAncap

Today I registered some hex names.

They're not just "fancy" but have use cases, one of which I just added to SupraNS. Such names are not novel, but they're rarely used and, in my opinion, interesting. So I'll write a few words about that.

Say you get a SNs name that consists of hexadecimal symbols ([0-9][A-F]). Example: b00b5.

Is that interesting and/or useful? Let's take a closer look.

Downsides of hex SNS names​

A "hex name" can be dangerous if you get autocorrected when entering such SNS names. For example, a payee may end up entering boobs.supra in StarKey wallet and send funds to the wrong address.

But that risk depends on the name and whether the "wrong" SNS exists (in many cases it would). Most names shouldn't get autocorrected (especially not b00b5). In some cases hex names will be less autocorrect-prone because they may not match common misspellings (thanks to the fact that they usually contain digits).

Another danger is visual confusion, where someone misreads 1abe1.supra and sends funds to label.supra. One workaround is to register the both SNS names. Another is to not make such mistakes (e.g. use a good font, or advise your counterparty to use copy-and-paste rather than view-and-retype when entering your SNS hex address).

Benefits of hex SNS names​

First, they're fancy and can be c001 (somebody snapped this one before me).

Second, they cay be practical. If you create a wallet address that matches your SupraNS name for easy address recognition (e.g. in SupraScan.io or API output), it's easier to use or develop. For example, I may have SNS b00b5.supra mapped to wallet address b00b57fef64e98e838eef231215dd0e7fa634d2fd70e75ae50df3a2959eba2f4.

Or, the proverbial Alice may be boob5 + a, which allows me up to 26 test addresses for easy testing:

alice.b00b5.supra # b00b5a7fef64e98e838eef231215dd0e7fa634d2fd70e75ae50df3a2959eba2f
jane.b00b5.supra  # b00b5j16c6bde199856efd1204ae840d21e3a26d0c501fb1521a0d9f7302934d
note

Note that StarKey won't create vanity addresses for you. Even if it could, it'd be slow for anything but 3 letters.

You'd have to generate such addresses outside and import their private key to StarKey. You don't need that for SupraScan.io, CLI or API use cases.

Other notes​

SNS names can be as short as 3 single-byte characters.

Long hex names are impractical: hard to read, hard to tyype. If you combine two "hex words" (say, b19b00b5), that's probably the upper limit. Three words are usually hard to read. And - isn't a hexadecimal symbol, so b19-b00b5 is possible in SNS but can't be mapped to a wallet address.

The last bit is related to those who want to use vanity addresses. Today it's still quite hard to generate them for 7 or more characters.

Therefore I recommend SNS length of hex names to be in the range 3-6 (characters). If you don't plan to use vanity addresses, then up to 10(-ish) may be fine, assuming you don't pick a lame name.

Where to buy or register hex names​

I registered a bunch of hex names according to the above considerations: most are 3-6 characters and therefore suitable for pairing with vanity addresses. Some are longer, but I registered only those long hex names that I really liked.

You can find them by sorting the table of available names by Comment and looking for comments that contain "hex".

Some good ones (including 3 letter names) are still available through Dexlyn's SupraNS dApp.

Update (March 25, 2025)​

SupraNS names are now transferable directly from Dexlyn's dApp UI.

There's no marketplace yet, but we have been gifting and exchanging names in a trusted way.

warning

Make a backup of your StarKey pass phrase. Like with other non-custodial walles, there's no other way to get access.

  • Network: Ethereum
  • Contract: paste the WXX contract address you got earlier

The rest will be auto-populated.

  • Click on All Networks at the top
  • Click on Testnet and then again on Mainnets and return to wallet

That makes all zero-balance coins appear in view. They may disappear again, but at least you can copy an addresses to fund it and make it a non-zero balance address, or repeat the workaround to view them again at any time.

Good Samaritans of Supra Network

"And who is my neighbor?"

Situation​

Supra launched and the process of token distribution was a mess. (The network itself is promising.)

Among the problems, one of the small ones was that many "airdrop" users weren't able to "claim" their $SUPRA because they couldn't pay for the transaction fee required to do that.

Of course, that's not really a problem: open a CEX account, buy $SUPRA and send it to your address. 15 minutes of easy labor.

But why work 15 minutes, solve your problem and put yourself in a position in which you can also help your fellow users (1), when you can beg for free in Supra Discord instead?

(1) By this I mean residents of countries who cannot open accounts in currently available CEXes. None of the current tax haven CEXes accept US account holders, for example.

Good Samaritans​

From an economics perspective, there are several interesting things about this.

Discord admins​

Discord channel admins allowed begging, which is commonly banned everywhere because it doesn't add any value. It is also counterproductive because it adds noise, and doesn't help the project (whereas sending uses to CEXes does).

Community members​

First, I was suprised that relatively few people in Discord complained, but I think I know why: nobody bothers to read anything but their own threads and the lax moderation makes that worse becuase there's simply too many stupid posts in the channel (#general on the Supra Discord). At the same time, many are mad about the problems in the claim process, so the administrators can't be as strict as they should be.

Second, some users, like farmsenth.eth on X, offer to send 1 $SUPRA for "free" to anyone, to help them claim tokens.

The Supra guys reposted that, but I commented about this on X saying it's a terrible idea that doesn't help Supra at all. Later I noticed a change - they started telling users to go to a CEX and buy $SUPRA - so looks like my feedback had some effect.

I forgot to mention that feeding beggars just makes the problem worse because those who would reluctantly open a CEX account would decide to skip that and beg instead, seeing how it's allowed and works.

Third, there were two groups of people who didn't feed the beggars (as much):

  • Zero-interest lenders: those who offered to lend 1 $SUPRA under the condition that the borrower pays back after they claim
  • Predatory profiteers: those who offered to lend at a profit. I belong to this group.

Who's the bad guy here?​

People like farmsenth.eth are the worst, for reasons explained above and in the tweet. In this group, the Discord account souravrockz (maybe the same guy, I haven't looked) is particularly active, using free $SUPRAs as a marketing tool for gathering followers for his airdrop platform. If that's not definition of spamming, I don't know what is. But they let him anyway. farmsenth.eth does the same on X, buys followers while making CEX partners lose customers they probably justifiably expected to win when they invested in marketing and launch activities.

The lenders are not as bad, although they too do the same disservice to the network because they decrease liquidity by helping users stay "inactive" as far as CEXes are involved.

note

Note that users who claim tokens without opening a CEX account cannot actually do anything with them.

There's no staking on Supra as of now and there are no DEXes for $SUPRA either, so there's no reason to help them - they cannot possibly do anything with the coins if they do not create a CEX account.

The only difference between helping them and not helping them is if you help them they won't need to buy few more $SUPRA - they can dump the tokens without any fiat investment in the process.

Typical example: buying 1,000 and then immediately selling 790 $SUPRA leaves you with 210 $SUPRA that you can send to your wallet. Early on (within a week from the launch) that was less than a $2 in "additional investment". But the pikers were relentless. Why buy 210 when I can beg for 1?

How many supras in gas do I need to be able to claim bybit? It asks for a minimum purchase of 1000 supras. That is too much. Is there no way to get gas elsewhere? ~ beggar from Supra Discord

Profiteers such as I are less bad than the lenders because by asking to be compensated we discourage begging (since not all beggars want to pay more back).

I would prefer that beggars be banned or kicked so that there's no begging and everyone goes to CEX so that only a small minority (Americans and Europeans, mostly) asks for help. But begging is going on and the first two groups doing their thing, so I decided to participate using what I belive is the least bad approach.

Did it work?

  • Yes, when no one from the other two groups (donors and lenders) happens to be online
  • No, when someone from one or both groups offers to help out for free

Predatory lending?​

Is 500% interest per day predatory lending?

No, because it takes me minutes of work (copy, login, send, messaging) to do it and the "profit" was 8 $SUPRA (approximately 15 cents at the time) per customer.

That's not even US$ 9 per hour, so I worked for less than a minimum wage in the West while having capital at risk. Sorry, but I'm not sorry!

Results​

Another fun aspect of this has been profitability.

I've sent 2 $SUPRA to 4 people, each of whom agreed to pay me back 10 $SUPRA.

Only two paid back anything. From what I could tell, one was Chinese and one was American.

note

The user from China went above & beyond: he asked for 1 $SUPRA. I sent him 2 just to be sure he doesn't get stuck, and he sent me back 20 $SUPRA!

Maybe he assumed the original 1-for-10 deal meant I expected 20 in exchange for 2 and he didn't want to make a big deal out of it. Real gentleman!

I later sent him 9 $SUPRA to stick to the original deal (1 in exchange for 10).

These two wallet owners weren't as nice. They agreed to receive 2 and pay back 10 (approximately $0.015 and $0.15, respectively, at the time) and failed to deliver:

  • 0x699c017fe9b7112a98e58d9896eee3c8fc57ca985f161527c1aeeb491c2093b7
  • 0xf1610d90bbc83b967a9b993f05de1e0eca61d2bf45003d840cd3cd0a896638e0

They didn't tell me why they didn't pay back, informed me why they couldn't, or told me about some delay or anything like that.

From my side, I didn't ask them if they maybe "forgot" about the deal, but I'll gladly link their wallets here.

Another guy dropped out "in presales stage" - he wanted just 1 $SUPRA (not 2, which was stupid since the assumption was he'd return both anyway), but didn't commit to returning any, so I disqualified him.

The spammers and lenders did another 20-30 transactions that I saw (and probably 10x more when I wasn't looking), so one of my conclusions is certainly that people who really needed help were few.

The rest were simply lazy or outright scammers. Some probably didn't even have any $SUPRA to claim - they just wanted to profit from begging.

Since then $SUPRA went up a lot (it's $0.3 now), so I'd probably ask 2 $SUPRA in return for 5 $SUPRA at current prices. I'd ask for less, but given these levels of honesty it's hard to not lose money saving the world...

I expect that aidrop spammer guy will soon stop giving free $SUPRA for nothing and - given these lending losses - I think the not-for-profit lenders will also go away.

Only profiteers will remain in the game. The only way lending can always work.
